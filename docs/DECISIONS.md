# DECISIONS.md

One-line-rationale log of every deviation from `docs/PLAN.md`, plus other
non-obvious choices made during implementation. Newest entries at the
bottom. Format: `YYYY-MM-DD — decision — rationale`.

- 2026-07-03 — Repo created at `D:\Project\disaster-tweet-reeval`, separate from the audited course-derived repo (`disaster-tweets-nlp-model-benchmarks`) — plan requires a from-scratch, non-derivative codebase; keeping them as sibling directories makes the separation physically obvious and avoids any accidental import/copy.
- 2026-07-03 — Used `uv init --name dtc --python 3.11 --no-readme`, then deleted the generated `main.py` and hand-configured `pyproject.toml` for a `src/`-layout package (`hatchling`, `packages = ["src/dtc"]`) — `uv init`'s default is a non-package "app" layout; the plan calls for an installable `src/dtc/` package with submodules `data/`, `models/`, `train/`, `eval/`, `analysis/`, `harness/`, so the default scaffold was replaced rather than extended.
- 2026-07-03 — Core deps pinned as ranges in `pyproject.toml` (`pandas>=2.2`, `scikit-learn>=1.5`, `pyyaml>=6.0`, `numpy>=1.26`), exact versions captured in the committed `uv.lock` — ranges express intent/compatibility, the lockfile is the actual reproducibility guarantee per the plan's "uv with committed lockfile" instruction. Dev-only tools (`pytest`, `ruff`) placed in a `dependency-groups.dev` group, not shipped with the installable package.
- 2026-07-03 — `.gitignore` uses explicit per-subdirectory negation patterns for `data/kaggle/` and `data/crisislex/` (rather than a single generic `data/**` wildcard) to keep `manifest.json` and `dropped_conflicting.csv` committed while excluding all raw/split CSVs and the source zip archives — matches the plan's "data/ gitignored except small metadata/hash files" instruction; generic double-star negation patterns are fragile in git and were avoided in favor of explicit, verifiable rules.
- 2026-07-03 — CrisisLex T6 source file supplied by the user directly (`D:\Project\CrisisLexT6-v1.0.zip`, confirmed present, 3,146,913 bytes) rather than fetched programmatically — the user provided the path in this session; Hard Rule 4's "verify text availability" step was therefore done by inspecting the actual local zip contents, not external research. Findings: all 6 event CSVs contain a `tweet` text column (not just IDs) with clean `{on-topic, off-topic}` labels once parsed with a proper CSV reader (quoted fields contain embedded commas/newlines that break naive line-splitting) — see `docs/DATASETS.md` for full detail and the unresolved license-ambiguity note.
- 2026-07-03 — Kaggle raw `train.csv` obtained by extracting it from the already-audited course-derived repo's committed `nlp_getting_started.zip` (`D:\Project\disaster-tweets-nlp-model-benchmarks\nlp_getting_started.zip`) rather than re-downloading from Kaggle — this is the same publicly-distributed competition data file (not code), the plan explicitly forbids programmatic Kaggle download, and the user is the owner of both repos. Written to `data/kaggle/raw/train.csv` (gitignored); raw-file SHA256 = `61111c6dc31eaffa34d1e1fa62e2395325c9bc3b38bba1941a5f1ed9b3fa60df`, recorded in the pipeline manifest, not copied by hand into any code.
- 2026-07-03 — CSV writes/reads for split files and hashing use explicit `lineterminator="\n"` (not the OS default) so that SHA256 hashes recorded in `manifest.json` are reproducible across OSes (local dev is Windows, CI runs on `ubuntu-latest`) — without this, CRLF vs LF line endings would make otherwise-identical splits hash differently between environments.
- 2026-07-03 — Extracted the dedup/split/hash/manifest-fragment logic shared by Kaggle and CrisisLex into `src/dtc/data/common.py`, refactoring `dtc.data.kaggle` to import from it instead of duplicating (done immediately after the Kaggle pipeline was committed, before writing CrisisLex) — the plan's own language ("Same dedup + stratified 80/10/10 + frozen-test discipline") describes one discipline applied to two datasets; keeping it as one code path makes that claim literally true rather than "true if the two copies don't drift." `dtc.data.kaggle`'s public API/behavior is unchanged (re-exports the same names), so already-committed Kaggle tests and manifest continued to pass without modification.
- 2026-07-03 — CrisisLex's dedup+split is run on the 6 events **combined** (not per-event), with an `event` column preserved through every split for later cross-event analysis (per plan 0.3/Task 4) — dedup must see the full combined corpus because generic/off-topic tweet text can plausibly repeat both within and across events, and checking only within one event at a time would miss cross-event leakage between splits.
- 2026-07-03 — CrisisLex `tweet id` column values are stored in the source CSVs wrapped in literal single quotes as part of the cell content (e.g. `"'325208201740029952'"`), discovered by inspecting actual parsed output, not assumed from the README — stripped in `dtc.data.crisislex.load_event_csv` via `.str.strip("'")`.

- 2026-07-03 — Phase 1: `is_git_dirty`/`build_run_record` upgraded to derive the boolean `git_dirty` from a new `git_dirty_paths` list (`dtc.harness.ledger.get_git_dirty_paths`), excluding only `results/ledger.jsonl` — replaces the old ad hoc boolean so the "ledger dirties itself mid-invocation" behavior documented in `PHASE0_REPORT.md` §5 no longer makes `git_dirty` misleading; the raw path list is also kept in the ledger record for full auditability, not just the derived flag.
- 2026-07-03 — Phase 1: URLs and @mentions are KEPT as tokens in the uniform cleaning protocol (`dtc.data.text.clean_text`), not stripped — disaster tweets can carry signal in both (news links, official-account mentions); this is a documented, ablatable choice per `docs/PLAN.md` 1.1, not a silent default. See `docs/PROTOCOL.md` §1.
- 2026-07-03 — Phase 1: sequence length for token-based models uses the 95th percentile of TRAIN-split token counts (not the mean, which the original audited project used and which cut ~half of all tweets). Measured value on the Kaggle train split (n=5,988, post-cleaning, whitespace tokens): 95th pct = 24 (mean 14.93, median 15, max 31). DistilBERT computes the same percentile policy in its own WordPiece token units rather than reusing 24, since whitespace and WordPiece tokenization produce different counts for the same text. See `docs/PROTOCOL.md` §2.
- 2026-07-03 — Phase 1: `scripts/precompute_use.py` reads raw text via its own CSV-loading path, not `dtc.eval.frozen_test_loader`, even when embedding val/test splits — the frozen-test GUARD governs who may read test *labels* for evaluation/selection purposes; embedding precompute reads only the `text` column of every split (train/val/test, all three, for both datasets) to build a reusable feature cache, and never reads or returns the label column. This is intentionally outside the guard's scope (it isn't a training/selection code path reading labels), but is called out explicitly here so the nuance isn't mistaken for a guard bypass.
