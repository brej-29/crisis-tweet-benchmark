- 2026-07-03 — Phase 1: `is_git_dirty`/`build_run_record` upgraded to derive the boolean `git_dirty` from a new `git_dirty_paths` list (`dtc.harness.ledger.get_git_dirty_paths`), excluding only `results/ledger.jsonl` — replaces the old ad hoc boolean so the "ledger dirties itself mid-invocation" behavior documented in `PHASE0_REPORT.md` §5 no longer makes `git_dirty` misleading; the raw path list is also kept in the ledger record for full auditability, not just the derived flag.
- 2026-07-03 — Phase 1: URLs and @mentions are KEPT as tokens in the uniform cleaning protocol (`dtc.data.text.clean_text`), not stripped — disaster tweets can carry signal in both (news links, official-account mentions); this is a documented, ablatable choice per `docs/PLAN.md` 1.1, not a silent default. See `docs/PROTOCOL.md` §1.
- 2026-07-03 — Phase 1: sequence length for token-based models uses the 95th percentile of TRAIN-split token counts (not the mean, which the original audited project used and which cut ~half of all tweets). Measured value on the Kaggle train split (n=5,988, post-cleaning, whitespace tokens): 95th pct = 24 (mean 14.93, median 15, max 31). DistilBERT computes the same percentile policy in its own WordPiece token units rather than reusing 24, since whitespace and WordPiece tokenization produce different counts for the same text. See `docs/PROTOCOL.md` §2.
- 2026-07-03 — Phase 1: `scripts/precompute_use.py` reads raw text via its own CSV-loading path, not `dtc.eval.frozen_test_loader`, even when embedding val/test splits — the frozen-test GUARD governs who may read test *labels* for evaluation/selection purposes; embedding precompute reads only the `text` column of every split (train/val/test, all three, for both datasets) to build a reusable feature cache, and never reads or returns the label column. This is intentionally outside the guard's scope (it isn't a training/selection code path reading labels), but is called out explicitly here so the nuance isn't mistaken for a guard bypass.
- 2026-07-03 — Phase 1 Protocol A (E2) hyperparameters were pulled from `PROJECT_AUDIT.md` (sibling repo `disaster-tweets-nlp-model-benchmarks`, consulted fresh this session, not copied from earlier context) rather than guessed, per the task's "sensible defaults mirroring the audited project where they exist" instruction: `Embedding(10000, 128)` for every from-scratch model, `LSTM(64)`/`GRU(64)`/`Bidirectional(LSTM(64))`/`Conv1D(filters=32, kernel_size=5)` architectures, no dropout anywhere, `epochs=5`, Adam defaults (lr=0.001), no early stopping, and — notably — `TfidfVectorizer()`/`MultinomialNB()` with ALL sklearn defaults (no `max_features` cap at all, unlike this project's own 10000 default). The audit also revealed the original notebook's markdown claims a "Logistic Regression" model that was never actually executed (only MultinomialNB was); `tfidf_logreg` and `distilbert_finetune` therefore have NO original equivalent to mirror and use documented plain defaults instead, noted as such in their `configs/protocol_a/*.yaml` files. USE's original architecture (`Dense(64, relu) -> Dense(1, sigmoid)`, frozen encoder, no dropout) matches this project's own `use_frozen` default `hidden_size=64` already, confirmed rather than assumed.
- 2026-07-03 — Phase 1: `scripts/precompute_use.py` gained an `--extra-csv` option (and `main(..., extra_csv_paths=...)`) so Protocol A's `use_frozen` runs have full USE-embedding cache coverage over the RAW (non-deduped) Kaggle csv, not just the Protocol B prepared splits — the ~165 rows Protocol B's dedup policy drops would otherwise raise `KeyError` in `UseFrozenModel` if they land in Protocol A's 90/10 split. Cache entries are keyed by text SHA256 regardless of source, so this is purely additive to the existing per-dataset cache, not a second cache.
- 2026-07-03 — Introduced a `stage` field on every ledger run record (values used so far: `"tuning"`, `"E1"`, `"E2"`, `"E3"`, or `None`) rather than adding a separate `experiment` field alongside it — one field fully identifies "what this run is for" without redundant/potentially-inconsistent dual bookkeeping. `docs/PLAN.md`'s "ledgered with stage: tuning" language is satisfied literally, and scripts/run_matrix.py reuses the same field to tag E1/E2/E3 runs and to compute its skip-key.
- 2026-07-03 — `dtc.models.torch_common.fit_with_early_stopping[_generic]` gained `patience: int | None` (None disables early stopping) and `restore_best_weights: bool` (False keeps final-epoch weights instead of the best-val-loss checkpoint) — needed so Protocol A's "no early stopping, fixed N epochs" replication (docs/PLAN.md 1.3) could reuse the exact same model classes and training loop as Protocol B, rather than forking a separate no-early-stopping implementation per model.
- 2026-07-04 — Bug found during real smoke verification (Task 8): `scripts/precompute_use.py` was caching USE embeddings keyed by SHA256 of the RAW (uncleaned) CSV text, while `dtc.models.use_frozen` hashes CLEANED text (`dtc.data.text.clean_text`) at fit/predict time per docs/PROTOCOL.md §4 ("USE consumes cleaned raw strings") — a real KeyError on the first live `run_matrix.py --smoke --only tuning` invocation, not a hypothetical. Fixed by having `precompute_use.py` clean text before hashing/embedding, matching the consumer. The stale (wrongly-keyed) `data/kaggle/use_embeddings/` cache was deleted and regenerated rather than left alongside the corrected files.
- 2026-07-04 — The crash above happened after 40 of 52 tuning-smoke runs had already succeeded and been ledgered (the crash was inside `use_frozen`'s `fit`, the 8th of 9 models in enumeration order); re-running `run_matrix.py --smoke --only tuning` after the fix correctly skipped those 40 and ran only the remaining 12 (`use_frozen` + `distilbert_finetune`) — an unplanned but genuine real-world demonstration of Task 6's resumability guarantee (a killed/crashed run leaves no partial ledger line and doesn't force redoing already-ledgered work), not just the synthetic fixture in `tests/test_run_matrix.py`.
- 2026-07-04 — Bug found while wiring up `configs/final/README.md` (Task 8): `scripts/select_configs.py`'s docstring claimed smoke runs are ignored, but `select_best_configs` only filtered on `stage == "tuning"`, with no smoke check — so once real (non-smoke) tuning runs eventually share the ledger with smoke tuning runs, a smoke config could still outrank/contaminate the real selection.
